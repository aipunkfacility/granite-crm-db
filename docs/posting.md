как можно автоматизировать рассылку через вотсап, со своего аккаунта? какие лимиты?

Автоматизация рассылки через личный аккаунт WhatsApp сопряжена с риском перманентного бана. Существует три основных пути:

### Способы автоматизации

1.  **Библиотеки (Self-hosted):**
    * **whatsapp-web.js (Node.js):** Эмулирует работу WhatsApp Web. Популярный и гибкий инструмент.
    * **Venom-bot:** Аналогичная библиотека на JavaScript с широким функционалом.
    * **Pywa / Pywhatkit (Python):** Простые решения для быстрой отправки, но менее стабильные для массовых задач.
2.  **Сторонние шлюзы (API Wrappers):**
    * **Green-API, Whapi, Maytapi:** Предоставляют готовый HTTP API для вашего личного номера (через QR-код). Платные, но избавляют от поддержки своей инфраструктуры.
3.  **Официальный API (WABA):**
    * **WhatsApp Business Cloud API:** Самый надежный метод. Не привязан к физическому телефону, нет риска бана за саму рассылку, но сообщения платные и требуют одобрения шаблонов.

### Лимиты и риски (Personal Account)

Официальных лимитов для личных аккаунтов нет, но работают алгоритмы антифлуда:

| Параметр | Рекомендация для личного аккаунта |
| :--- | :--- |
| **Количество** | Не более **50–100** сообщений в сутки новым контактам. |
| **Паузы** | Рандомные задержки между сообщениями (**20–60 секунд**). |
| **Контент** | Избегайте идентичных текстов (используйте спин-текст или ИИ). |
| **Главный риск** | Кнопка **«Пожаловаться»**. 3–5 жалоб подряд — мгновенный бан. |

### Шаг 1: Настройка Node.js воркера (WhatsApp)

Используйте `whatsapp-web.js`. Она лучше всего подходит для низкочастотных (10–20 в день) и персонализированных касаний.

```javascript
const { Client, LocalAuth } = require('whatsapp-web.js');
const sqlite3 = require('sqlite3').verbose();

// Подключение к вашей базе данных Granite
const db = new sqlite3.Database('./data/granite.db');

const client = new Client({
    authStrategy: new LocalAuth({ dataPath: './sessions' }),
    puppeteer: { 
        headless: true,
        args: ['--no-sandbox'] 
    }
});

client.on('ready', () => {
    console.log('WhatsApp воркер готов к работе');
    // Запуск проверки очереди каждые 15 минут
    setInterval(checkTaskQueue, 15 * 60 * 1000);
});

async function checkTaskQueue() {
    const query = `
        SELECT t.id, t.company_id, c.phones, tmp.body 
        FROM crm_tasks t
        JOIN enriched_companies c ON t.company_id = c.id
        JOIN crm_templates tmp ON tmp.name = 'wa_follow_up'
        WHERE t.status = 'pending' AND t.task_type = 'follow_up'
        LIMIT 1`; // Берем по одному для безопасности

    db.get(query, async (err, task) => {
        if (task) {
            // Эмуляция печати и отправка
            await sendMessage(task);
        }
    });
}
```

### Шаг 2: Python для управления логикой (AI Architect)

Python через `SQLAlchemy` или прямой `sqlite3` управляет воронкой и решает, кому пора писать в WhatsApp.

* **Логика:** Если `email_sent` был 5 дней назад и ответа нет, Python создает запись в `crm_tasks`.
* **Преимущество:** Вы можете использовать библиотеки ИИ (LangChain/OpenAI) для динамической генерации текста сообщения, чтобы каждое из 20 сообщений было уникальным.

### Шаг 3: Синхронизация статусов

Чтобы система работала как единое целое, Node.js после отправки должен обновлять базу, а Python — это видеть:

1.  **Node.js отправляет сообщение.**
2.  **Node.js пишет в `crm_touches`**: канал `wa`, направление `outgoing`.
3.  **Node.js обновляет `crm_contacts`**: `wa_sent_count += 1` и `last_wa_at = now()`.
4.  **Python-дашборд** (или FastAPI) подхватывает изменения и переводит лида в статус `second_follow_up`.

Telegram-бот в планах, как его интегрировать?

Для вашей воронки (Email → TG → WA) важно различать **Bot API** и **UserBot (MTProto)**. 

Обычный бот не может написать первым человеку, который его не запускал. Для холодного аутрича в Telegram вам нужен **UserBot** — скрипт, который управляет вашим личным аккаунтом.

### Технический стек

1.  **Библиотека:** * **Node.js:** `gramjs` (аналог `whatsapp-web.js` для Telegram).
    * **Python:** `Pyrogram` или `Telethon`. Для вашей архитектуры **Pyrogram** на Python — самый мощный и стабильный вариант для сессий.
2.  **Интеграция:** * Python-скрипт (UserBot) так же, как и Node-воркер для WhatsApp, опрашивает таблицу `crm_tasks`.
    * Если `channel = 'tg'`, скрипт берет номер телефона или username из `enriched_companies` и отправляет сообщение.

### Схема работы в Granite CRM

* **Инициация:** Python-логика видит, что email открыт, но ответа нет → создает задачу в `crm_tasks` для TG.
* **Отправка:** UserBot отправляет сообщение. Важно: Telegram жестко банит за ссылки в первых сообщениях. Пишите голый текст.
* **Синхронизация:** При получении ответа UserBot пересылает его вам в системный чат или обновляет статус в БД на `replied`.

### 1. Как Python узнает об открытии письма

Стандарт индустрии — **Tracking Pixel**. Это невидимое изображение размером 1x1 пиксель, которое подгружается с вашего сервера при открытии письма.

**Механика:**
1.  При отправке письма через Python (например, `smtplib` или `Resend`), вы генерируете уникальный `UUID` для этого касания.
2.  В HTML-тело письма вставляете тег:
    `<img src="https://your-crm-api.com/track/v1/pixel.png?id=UUID_ID" width="1" height="1" style="display:none;">`
3.  Ваш бэкенд (FastAPI/Astro) ловит запрос, извлекает `id`, находит лида в SQLite и обновляет поле `email_opened_at`.

> **Нюанс:** Современные почтовики (Apple Mail, Gmail) могут кэшировать картинки или блокировать их. Для 100% точности также отслеживайте клики по ссылкам в письме через редирект-прокладку.

Использовать основной номер для автоматизации — **опасно**. Алгоритмы WhatsApp и Telegram могут забанить аккаунт навсегда без возможности восстановления.

сервер должен быть постоянно в сети?

Да, для полноценной работы **Granite CRM v4** сервер (или устройство, выполняющее его роль) должен быть в сети **24/7**.

Вот почему это критично для вашей архитектуры:

### 1. Регистрация «Открытий» писем (Tracking Pixel)
Трекинг-пиксель — это запрос к вашему серверу. Если в момент, когда клиент открывает письмо, ваш сервер выключен, запрос уйдет в никуда (ошибка 404 или Timeout).
* **Результат:** Вы не узнаете, что клиент проявил интерес, и цепочка автоматизации (follow-up в мессенджеры) не запустится.

### 2. Прием ответов (Webhooks)
Когда клиент пишет вам в WhatsApp или Telegram, мессенджеру нужно куда-то доставить это событие, чтобы CRM перевела лида в статус «Ответил» и остановила автоматическую рассылку.
* **Результат:** Если сервер офлайн, бот может продолжить слать запланированные follow-up сообщения человеку, который уже ответил, что выглядит крайне непрофессионально.

### 3. Стабильность сессий (WhatsApp Web)
Библиотека `whatsapp-web.js` поддерживает активное соединение. Если интернет пропадет надолго, сессия может «протухнуть», и вам придется заново сканировать QR-код.

## Как обезопасить личный аккаунт (Технически)

Если всё же решите использовать основной номер, внедрите **«Предохранитель» (Kill-switch)**:

1.  **Проверка входящих:** Перед каждой автоматической отправкой скрипт должен проверять таблицу `crm_touches`. Если за последние 24 часа был **входящий** месседж от этого номера — автоматика блокируется для этого контакта.
2.  **Ручной режим:** Добавьте в CRM кнопку «Пауза для всех», чтобы мгновенно остановить воркеры, если вы сами зашли в WhatsApp и начали переписку.
3.  **Имитация «Сна»:** Скрипт не должен работать 24 часа. Настройте рассылку строго по рабочим часам региона (например, МСК 09:00–18:00). В остальное время сервер может быть в сети, но воркеры должны «спать».

**Резюме:** Для сбора статистики открытий сервер должен быть доступен извне 24/7. Для самой рассылки достаточно включать скрипты на пару часов в день.

напиши гайд по разработке функции поиска по номерам

Для реализации функции поиска (резолвинга) контактов по номерам телефонов в Telegram необходимо использовать **MTProto API** (через библиотеку `Pyrogram`), так как обычные боты не имеют доступа к телефонной книге пользователей.

Ниже представлен подробный гайд по интеграции этой функции в вашу **Granite CRM v4**, включая обновленный код воркера и стратегический промпт для дальнейшей разработки.

### Ключевые этапы разработки

1.  **Нормализация данных:** Приведение всех телефонов из БД к международному формату `E.164` (например, `79991234567`).
2.  **Метод ImportContacts:** Использование системного вызова Telegram для временного добавления номера в контакты аккаунта. Только так можно получить `user_id`.
3.  **Сбор метаданных:** После успешного резолвинга извлекается `username`, статус `was_online` (для оценки "живости" лида) и `bio` профиля.
4.  **Лимиты и паузы:** Telegram жестко ограничивает перебор номеров. Безопасный лимит — до 50-100 проверок в сутки с одного аккаунта с паузами в 15-30 секунд между запросами.

### Важные советы для вашей CRM:
* **Имя при импорте:** Я добавил `last_name=str(company_id)`. Это поможет вам в будущем понять, какой именно контакт в вашей телефонной книге ТГ соответствует ID в базе данных.
* **Очистка контактов:** Периодически (раз в неделю) нужно запускать скрипт, который удаляет импортированные контакты из аккаунта через `functions.contacts.DeleteContacts`, иначе список контактов превратится в "свалку" из тысяч записей.
* **Если TG не найден:** Как мы обсудили, в этом случае в поле `telegram` ставится `ABSENT`, что является сигналом для CRM переключить воронку на **WhatsApp**.

Полный гайд по автоматизации Granite CRM v4

Данный документ представляет собой подробное техническое руководство по интеграции и эксплуатации системы холодного аутрича в цепочке Email → Telegram → WhatsApp. Руководство адаптировано для работы из Вьетнама с учетом специфики часовых поясов и рисков блокировок.

1. Архитектурный стек и взаимодействие

Система построена на принципе разделения обязанностей (Separation of Concerns) между двумя языками программирования для достижения максимальной стабильности.

База данных: SQLite (режим WAL).

Зачем: WAL (Write-Ahead Logging) позволяет Python-скрипту читать данные, пока Node.js-воркер записывает отчет об отправке, без блокировки БД. Это критично для параллельной работы модулей.

Мозг (Python): * Функции: Парсинг сайтов, сегментация лидов (A/B/C), генерация персонализированных офферов через LLM и управление очередью задач crm_tasks.

Пример: Если модель видит, что компания занимается «памятниками из Габбро», она подставляет этот контекст в шаблон сообщения.

Транспорт (Node.js + WhatsApp-web.js): * Функции: Управление сессией WhatsApp через Puppeteer (headless-браузер). Node.js выбран из-за событийной модели, которая лучше справляется с длительными WebSocket-соединениями мессенджеров.

Спецкор (Python + Pyrogram): * Функции: Работа с MTProto API Telegram. Позволяет обходить ограничения обычных ботов и инициировать диалог первым от лица обычного пользователя.

2. Мониторинг Email (Tracking System)

Для перехода от «холодного» письма к мессенджеру нам нужен сигнал об интересе. Самый надежный способ — невидимый пиксель.

Техническая реализация:

Backend (FastAPI): Создается легковесный эндпоинт, например GET /api/v1/img/{tracking_id}.png. Сервер отдает реальный прозрачный пиксель с заголовком Cache-Control: no-store.

База данных: Каждое отправленное письмо регистрируется в таблице crm_touches. Поле tracking_id (UUID) связывает конкретную отправку с лидом.

Логика обработки: При обращении к пикселю сервер фиксирует IP-адрес и User-Agent. Если обращение не похоже на бота (проверка по базе известных IP почтовых сервисов), поле email_opened_at обновляется.

Умная фильтрация: Чтобы избежать ложных срабатываний от антивирусных фильтров (которые "прокликивают" ссылки перед доставкой), мы игнорируем открытия в первые 5 секунд после отправки.

3. Автоматизация WhatsApp (Node.js)

WhatsApp крайне чувствителен к паттернам автоматизации. Наша задача — имитировать поведение реального человека, который работает в CRM.

Правила безопасности (Anti-Ban):

Лимиты и прогрев: Начинаем с 5 сообщений в день, увеличивая до 15–20 к концу второй недели. Это создает историю «органического роста».

Эмуляция печати (Typing Simulation): Мы не просто "выплевываем" текст. Скрипт вызывает событие client.sendPresenceAvailable() и client.sendStateTyping(), затем делает паузу (length * 100ms) перед отправкой самого сообщения.

Рандомизация пауз: Между отправками разным клиентам устанавливается задержка random(300, 600) секунд. Это "разрывает" математическую последовательность, по которой алгоритмы Meta вычисляют ботов.

Вариативность текста: Каждое сообщение должно отличаться хотя бы на 10-15%. Используйте Spintax (синонимайзеры) или динамические вставки (имя, название города, тип камня).

4. Автоматизация Telegram (Python)

Telegram UserBot — мощный инструмент, но требует аккуратного обращения с API ID и Hash.

Ключевые особенности:

Валидация контакта: Сначала скрипт проверяет наличие username. Если его нет, используется поиск по номеру телефона через ImportContactsRequest.

Запрет ссылок: Отправка ссылки в первом сообщении «холодному» контакту — это 80% вероятность вечного бана аккаунта. Сначала задаем уточняющий вопрос по производству, получаем ответ, и только потом даем ссылку.

Интеллектуальный стоп-кран: Как только в Pyrogram приходит событие OnMessage от лида, скрипт немедленно ищет этого лида в БД и ставит флаг stop_automation = 1. Это предотвращает ситуацию, когда вы уже общаетесь лично, а бот шлет "дожимающее" письмо.

5. Безопасность и гигиена аккаунтов

Потеря основного номера телефона — это катастрофа для бизнеса. Поэтому защита — приоритет №1.

Двойной Kill-Switch: * Проверка БД: Перед каждым шагом проверяется отсутствие входящих сообщений.

Проверка времени: Скрипт не работает по выходным и ночью по МСК, имитируя график живого менеджера.

Активный прогрев (Human Engagement): Аккаунт должен участвовать в "белых" переписках. Рекомендуется вступить в пару профильных чатов по камнеобработке и изредка писать туда нейтральные сообщения, чтобы алгоритмы видели активность в группах.

Физическая изоляция: Идеальный вариант — старый Android-смартфон с вьетнамской SIM-картой, подключенный к питанию. Это дает «чистый» Device ID и легитимный IP-адрес (через мобильный интернет), что значительно снижает подозрения мессенджеров.

6. Инфраструктура и стабильность 24/7

Поскольку вы находитесь во Вьетнаме, а клиенты в РФ, сервер должен работать непрерывно.

Варианты реализации:

Cloudflare Tunnel (Argo): Позволяет вашему домашнему ПК во Вьетнаме принимать запросы из интернета (для трекинг-пикселя) без необходимости открывать порты на роутере или покупать белый IP. Это бесплатно и безопасно.

Гибридный облачный буфер: * Вы арендуете минимальный VPS в РФ за 300-500 рублей.

На нем висит только приемщик пикселей и вебхуков.

Ваш локальный "тяжелый" компьютер раз в минуту забирает накопленные события через простой API-запрос. Это гарантирует, что ни одно открытие письма не будет потеряно из-за мигания света во Вьетнаме.
